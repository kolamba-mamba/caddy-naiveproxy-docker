# NaiveProxy и Caddy: Docker-конфигурация

Репозиторий содержит спецификации для развертывания NaiveProxy на базе сетевого стека Chromium в контейнеризированной среде Docker.

---

## Техническое обоснование

NaiveProxy нейтрализует фингерпринтинг (идентификацию отпечатков) TLS-соединений посредством интеграции оригинального сетевого стека Chromium.

### Архитектурные характеристики

1. **Стек Chromium:** Обеспечивает полную идентичность трафика браузеру Chrome, предотвращая обнаружение системами DPI (Deep Packet Inspection) по TLS-отпечаткам.
2. **Устойчивость к активному зондированию (Probe Resistance):** Реализует маскировку прокси-сервиса под стандартный HTTPS-ресурс. При отсутствии валидной аутентификации сервер транслирует контент определенного маскировочного ресурса.
3. **Протоколы передачи:** Инкапсуляция данных в HTTP/2 и HTTP/3 (QUIC) внутри стандартных TLS-сессий.
4. **Заполнение трафика (Padding):** Рандомизация размера пакетов для предотвращения статистического анализа паттернов трафика.

### Стратегия Heavy Camouflage

При передаче значительных объемов данных (от 1 ГБ/сутки) применяется директива `reverse_proxy`. Данная конфигурация имитирует работу зеркала или CDN-узла для стороннего ресурса, обеспечивая логическую легитимность высокого сетевого трафика. Рекомендуется использование релевантных поддоменов (`mirrorsite.you-domen.com`, `static.you-domen.com`, `cdn.you-domen.com`).

---

## Принцип функционирования

### Маршрутизация трафика

1. **Локальный слушатель:** Клиент инициирует SOCKS5-интерфейс на адресе 127.0.0.1:1080.
2. **Селективное проксирование:** На VPS направляется только трафик, явно адресованный на локальный порт. Системный трафик вне данной маршрутизации передается напрямую.
3. **Инкапсуляция:** Локальный агент преобразует SOCKS5-запросы в HTTPS-трафик стека Chromium для передачи на удаленный сервер Caddy.

## Процесс развертывания

### 1. Подготовка инфраструктуры

- Наличие доменного имени с корректной A-записью.
- Обеспечение доступности портов 80/TCP, 443/TCP и 443/UDP. Использование стандартных портов обязательно для корректной имитации браузерного трафика и работы ACME-клиента.

### 2. Инициализация параметров (.env)

Определение значений в файле `.env`:

- `DOMAIN`: Целевое доменное имя, на которое будет выпущен SSL-сертификат.
- `EMAIL`: Адрес электронной почты для регистрации учетной записи в Удостоверяющем центре (например, Let's Encrypt). Используется для уведомлений о состоянии SSL-сертификатов.
- `PROXY_SITE`: Ресурс для реверс-проксирования и имитации тяжелого трафика.
- `USER1_NAME`, `USER1_PASS`: Параметры аутентификации NaiveProxy.

### 3. Управление доступом

Активация дополнительных учетных записей производится путем раскомментирования соответствующих блоков `basic_auth` в `Caddyfile`.

### 4. Запуск сервиса

```bash
# Приведение прав доступа к UID 1000 для корректной работы томов данных
sudo chown -R 1000:1000 .

# Инициализация сборки и запуск контейнеров
docker compose up -d --build
```

## Конфигурация клиента

### Загрузка

Бинарные файлы для всех платформ доступны на странице релизов: [NaiveProxy Releases](https://github.com/klzgrad/naiveproxy/releases).

### Спецификация config.json

```json
{
  "listen": "socks://127.0.0.1:1080",
  "proxy": "https://USER:PASS@DOMAIN",
  "log": "",
  "padding": true
}
```

### Определение параметров

- **log**: Путь к файлу логов. Пустая строка `""` отключает логирование. Используется для отладки соединения.
- **padding**: Активация случайного заполнения HTTP/2 пакетов (Traffic Padding). Критически важный параметр для защиты от анализа трафика по размеру пакетов и временным интервалам (Traffic Analysis). Должен быть включен (`true`) для обеспечения максимальной скрытности.

### Пример интеграции в Telegram Desktop

1. Навигация: Настройки -> Продвинутые настройки -> Тип соединения.
2. Выбор: Использовать собственный прокси.
3. Конфигурация: Тип SOCKS5, Хост 127.0.0.1, Порт 1080.
